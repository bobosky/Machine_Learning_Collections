    Hi,

    I feel quite strange the fact that the month, asset name and previous returns appear to be the most import features. Don't you thing this should be a sign for data leakage ? When splitting the data in train / test randomly it means that model will be able to access similar observations of same or future days in the train set. In your particular case you did not split randomly you took the last 20% as validation (which is better than randomly) but still involves data leakage as target metric is calculated over 10 days. So you will most likely perfectly predict the first 10 days of the validation test. So let's say model finds a similar observation during same month (month feature) of the same asset (asset code) but 1/2 days after (or 1/2 day before which is also a problem because target metric is calculated over the next 10 days), he will be able to use it as a prediction directly. Is there something that prevent this situation ?

    Maybe I misunderstood because it's something I see everybody is doing,

    An alternative to avoid this problem would be not to split randomly but using the sklearn time_serie cross validation function : http://scikit-learn.org/stable/modules/generated/sklearn.model_selection.TimeSeriesSplit.html#sklearn.model_selection.TimeSeriesSplit

    Hi,

    I feel quite strange the fact that the month, asset name and previous returns appear to be the most import features. Don't you thing this should be a sign for data leakage ? When splitting the data in train / test randomly it means that model will be able to access similar observations of same or future days in the train set. In your particular case you did not split randomly you took the last 20% as validation (which is better than randomly) but still involves data leakage as target metric is calculated over 10 days. So you will most likely perfectly predict the first 10 days of the validation test. So let's say model finds a similar observation during same month (month feature) of the same asset (asset code) but 1/2 days after (or 1/2 day before which is also a problem because target metric is calculated over the next 10 days), he will be able to use it as a prediction directly. Is there something that prevent this situation ?

    Maybe I misunderstood because it's something I see everybody is doing,

    An alternative to avoid this problem would be not to split randomly but using the sklearn time_serie cross validation function : http://scikit-learn.org/stable/modules/generated/sklearn.model_selection.TimeSeriesSplit.html#sklearn.model_selection.TimeSeriesSplit
